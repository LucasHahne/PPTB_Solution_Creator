Deployment aborts: “Failed to retrieve MetadataId from response. The OData-EntityId header was missing.”

A technical post-mortem of why table and column creation aborted mid-deployment in the Solution Creator tool, even though the records were successfully created in Dataverse — and how the orchestrator was made resilient and idempotent.

Executive Summary

During a solution deployment, the tool issues a sequence of Dataverse metadata operations — create publisher, create solution, create tables, create columns, create relationships, publish. Two of these calls, createEntityDefinition (table) and createAttribute (column), began throwing: 

Error invoking remote method 'dataverse.createAttribute':
Error: Create attribute failed: Failed to retrieve attribute MetadataId
from response. The OData-EntityId header was missing.

The critical insight: the table and the column were actually created successfully in Dataverse. The error was raised after the write, by the Power Platform ToolBox host while trying to read the new record's MetadataId out of the HTTP response. Because the host throws, our orchestrator treats the whole step as failed and aborts — leaving a partially built solution and a confusing error.

Resolution in one line
We detect this specific post-write failure, recover the real MetadataId by looking the just-created object up by its logical name, and continue. We also made every step idempotent so re-running a partial deployment is always safe.

1 Symptoms

The failure was observed twice, on two different operations:

Run Operation What the user saw Reality in Dataverse
First createEntityDefinition Primary column reported created, then “The OData-EntityId header was missing”; no further columns created. Table and its primary name column existed.
Second createAttribute First whole-number column created, second one aborted with the same header error. Both whole-number columns existed; only the third was never attempted.
Key tell
The records always existed in the environment afterwards. That rules out a malformed payload or a permissions problem — the write half of the request succeeded every time. The fault was entirely in the response-reading half.

2 Architecture Context

The tool runs as a sandboxed web app inside an <iframe> hosted by the Power Platform ToolBox desktop app. The tool never talks to Dataverse directly. Instead it calls window.dataverseAPI.*, which marshals the request across the iframe→host boundary (Electron IPC). The host performs the authenticated HTTP call to the Dataverse Web API and marshals a result back. 

Tool (iframe)                Host (Electron)              Dataverse Web API
─────────────                ───────────────              ─────────────────
dataverseAPI                 ipc handler                  POST /api/data/v9.x/
 .createAttribute(...)  ──▶  'dataverse.createAttribute'  EntityDefinitions(..)/Attributes
                                       │                            │
                                       │   HTTP 204 No Content      │
                                       │   OData-EntityId:   ◀─┘
                                       │
                          reads MetadataId from the
                          OData-EntityId response header
                                       │
   { id } / throw  ◀──────────────────┘

When a metadata record is created, Dataverse returns 204 No Content with no body; the new record's GUID is delivered only in the OData-EntityId response header. The host parses the GUID out of that header to populate the { id } it returns to the tool.

3 Root Cause

3.1 The header the host depends on was absent

The host's createEntityDefinition / createAttribute implementation has a hard dependency on the OData-EntityId response header to extract the new MetadataId. In this environment the header was not present on the response, so the host threw before it could return an id. Common reasons the header is absent on an otherwise-successful metadata POST include:

  • The response was an empty 204 whose header set differed from what the host expects (gateway / proxy header stripping, casing, or exposed-header filtering).
  • The host attempts to read the header in a way that does not match how it was returned for these particular metadata endpoints.

From the tool's perspective the cause is external and not fixable by changing the payload — but it is detectable and recoverable.

3.2 Why the record nonetheless exists

The Dataverse server commits the metadata change and returns 204 Created/No Content before the header is ever read by the host. The header-read step is purely a client-side convenience to surface the new GUID. Therefore the exception is raised strictly after the durable write — the table/column is already persisted. This is why every failed step still showed up in the maker portal.

The real bug in our tool
Our orchestrator treated “host could not read the id” as “the operation failed,” aborting the deployment. The two are not the same. A post-write read failure must not be fatal when the write itself succeeded.

4 Impact

  • Deployments aborted non-deterministically at the first table or column whose response lacked the header.
  • Partial, inconsistent solutions were left in the environment (table created, only some columns created).
  • Retries made it worse before the fix: re-running hit “object already exists” errors on the records that had in fact been created, so there was no clean path forward.

5 The Fix

The fix has four parts, all in src/services/metadataService.ts and src/services/deploymentOrchestrator.ts.

5.1 Precisely detect the post-write failure

We only want to recover from this error — not mask genuine failures. A narrow matcher keys on both signature phrases (“MetadataId” and “OData-EntityId”), which covers both the entity and attribute wording.

src/services/metadataService.ts 
function isMissingEntityIdHeaderError(error: unknown): boolean {
  const message = error instanceof Error ? error.message : String(error);
  return /MetadataId/i.test(message) && /OData-EntityId/i.test(message);
}

5.2 Recover the real MetadataId by lookup

When the matcher fires, we ask Dataverse for the object we just created and read its MetadataId directly — turning the opaque failure into a successful, fully-correct result. Tables are looked up with getEntityMetadata; columns with getEntityRelatedMetadata using the Attributes(LogicalName='…') path.

src/services/metadataService.ts 
export async function createColumn(
  entityLogicalName: string,
  definition: Record<string, unknown>,
  solutionUniqueName: string,
  columnLogicalName?: string,
): Promise<string> {
  try {
    const result = await window.dataverseAPI.createAttribute(
      entityLogicalName, definition, options(solutionUniqueName),
    );
    return result.id;
  } catch (error) {
    // The write succeeded; the host just couldn't read the id header.
    if (columnLogicalName && isMissingEntityIdHeaderError(error)) {
      const recoveredId = await findAttributeMetadataId(
        entityLogicalName, columnLogicalName,
      );
      if (recoveredId) return recoveredId;
    }
    throw error;
  }
}

The table path (createTable) uses the same try/recover/rethrow shape, recovering via findEntityMetadataId.

5.3 Make every step idempotent

So a partially-completed deployment can always be safely re-run, the orchestrator now checks for existence before creating, and skips what already exists — for the publisher, the solution, each table, and each column.

src/services/deploymentOrchestrator.ts 
// Tables: skip those that already exist.
for (const entity of project.tables) {
  const logicalName = tableLogicalName(prefix, entity);
  const existingId = await findEntityMetadataId(logicalName);
  if (existingId) {
    log('info', `Table "${entity.displayName}" already exists — reusing it.`);
    continue;
  }
  /* ...create with recovery... */
}

// Columns: skip those already present on the table.
const existingColumns = await listColumnLogicalNames(logicalName);
for (const field of columns) {
  const columnLogicalName = buildLogicalName(prefix, field.schemaName);
  if (existingColumns.has(columnLogicalName)) {
    log('info', `Column "${field.displayName}" already exists — skipping.`);
    continue;
  }
  /* ...create with recovery... */
}

5.4 Bonus: a structured-clone guard

A related earlier failure was the opaque “An object could not be cloned” raised when a non-serializable value crosses the iframe→host postMessage boundary. We hardened against it by (a) using literal @odata.type strings instead of a host helper whose return value was not guaranteed clone-safe, and (b) verifying each payload with structuredClone before sending, so a bad value names itself instead of producing a cryptic error.

src/services/deploymentOrchestrator.ts 
function assertCloneable(label: string, payload: unknown): void {
  try {
    structuredClone(payload);
  } catch (error) {
    console.error(`[Solution Creator] ${label} is not structured-clone-safe:`, payload, error);
    throw new Error(`The ${label} payload contains a value that cannot be sent to Dataverse.`);
  }
}

6 Behaviour After the Fix

With recovery + idempotency, a single deployment now completes through the header anomaly, and any retry resumes cleanly:

✓
Create table → header missing Recovered MetadataId via getEntityMetadata; deployment continues instead of aborting.
✓
Create column #1 → header missing Recovered via Attributes(LogicalName='…') lookup; proceeds to next column.
✓
Re-run after a partial deploy Existing table and columns are detected and skipped; only missing items are created.
✓
Publish customizations Deployment finishes; all columns present.
Verified
User confirmed (“Works”) that all three whole-number columns and the table now deploy successfully. Project builds clean (tsc + vite, exit 0).

7 Engineering Takeaways

Principle Why it mattered here
Separate “write failed” from “read failed” A post-commit response-parsing error must not be treated as a failed mutation. We recover the result rather than aborting.
Design metadata writes to be idempotent Schema operations are not transactional and cannot be rolled back. Existence checks make every step safe to repeat.
Narrow your error matching Recovery is gated on a specific signature so genuine errors (bad payload, permissions, duplicates) still surface.
Validate IPC payloads at the boundary structuredClone guards turn opaque “could not be cloned” failures into named, actionable errors.
